> Status: code-ready (PR-1 + PR-2 merged, binaries hot-reloaded, env applied).
> The actual L2 rerun is an operator step because it depends on a long-running
> TTS+LLM job over chapter 2 of episode 143 (job 154). The agent has produced
> all code, tests, configuration and observability hooks needed; this document
> is the operator's checklist.

## Inputs

- Episode 143 → chapter 2 → job 154.
- Feature flags (all already applied in `.env`):
  - `SEGMENT_AGENT_ENABLED=true`
  - `JUDGE_VETO_DRIFT_RETRY=true`
  - `JUDGE_VETO_MIN_SCORE=0.95`
  - `DUBBING_PLAN_ENABLED=true`
  - `ENSEMBLE_RETRANSLATE_ENABLED=true`
  - `ENSEMBLE_MAX_PER_SEGMENT=2` (raised from 1 in PR-1 B3)
  - `RETRANSLATION_MAX_ATTEMPTS=30` (raised from 10 in PR-2 T2)
  - `SEGMENT_AGENT_ALLOW_SPLIT=false` (deliberately off)

## Step 1 — Trigger the rerun

The UI's "重跑" button hits `POST /jobs/154/retry` which calls
`Service.RetryJob` → `retryJobWithHint(hint=nil)`. We verified
`RetryJob` does NOT check `episodes.rework_status`, so the
`escalated_oscillation` halt on episode 143 will NOT block this rerun.

If the operator prefers CLI:

```powershell
# Verify current job state first
curl http://localhost:8080/jobs/154 | ConvertFrom-Json | Select status, current_stage

# Trigger retry
$body = @{ stage = "tts_duration" } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri "http://localhost:8080/jobs/154/retry" -Body $body -ContentType "application/json"
```

## Step 2 — Watch the agent decisions live

```powershell
# Tail worker logs filtered to agent_decision events for this job
docker logs -f holodub-worker-1 2>&1 | Select-String "job_id=154|agent_decision"
```

The 4 KPIs you want to track in real time (delta vs the previous run of
the same chapter):

1. **`use_ensemble=true` frequency** — expected to increase (PR-1 B2's
   abs-drift trigger fires on long high-drift segments that were
   previously stuck).
2. **`decision=accept reason=judge_veto_drift` count** — expected to
   decrease (PR-2 T1 tightened the VETO bands; same segments will
   now go through ensemble + retranslate first).
3. **`decision=accept reason=within_threshold` count** — expected to
   increase (PR-2 P1's adaptation strategy should help the LLM
   close the gap more often within the tight band).
4. **DUBBING_PLAN parse-recovery rate** (`holodub_llm_recovered_parse_total{operation="dubbing_plan"}`) —
   should be ≤ 1 across the chapter; if it's growing fast, the prompt
   constraint added by PR-1 B1 didn't stick and we need to inspect
   the raw tool args.

## Step 3 — Capture metrics + run baseline diff

After the rerun completes, snapshot both runs into JSON:

```powershell
# Pre-rerun baseline (already captured if you have it; otherwise pull
# from the existing per-job baseline file generated by the existing
# pipeline). For a fresh capture, hit the job metrics endpoint:
python tests/quality/run_baseline_diff.py collect --job-id 154 --out tests/quality/baselines/job154-pre.json

# After rerun:
python tests/quality/run_baseline_diff.py collect --job-id 154 --out tests/quality/baselines/job154-post.json

# Compare
python tests/quality/run_baseline_diff.py diff `
  --baseline tests/quality/baselines/job154-pre.json `
  --current tests/quality/baselines/job154-post.json
```

The differ uses the harness from PR-14 — KPIs that regress ≥ 5% trip
a `WARN`, ≥ 20% trip a `FAIL`.

## Step 4 — Decision point

| Outcome | Action |
|---|---|
| All 3 quality KPIs improve, cost < +15% | Keep flags on, schedule 79min L3 baseline run |
| Any quality KPI regresses ≥ 20% | Roll back PR-2 prompt + VETO tightening; keep PR-1 bug fixes |
| Ensemble triggers > 15% of segments OR job cost > $3 | Set `ENSEMBLE_MAX_PER_SEGMENT=1` and rerun |
| `holodub_llm_recovered_parse_total{operation="dubbing_plan"}` > 5 in the chapter | Investigate raw tool args; the prompt fix didn't take |

## Rollback procedure (if KPIs regress)

Each PR is independently revertable:

```powershell
# PR-2 prompt/T1/T2 rollback (revert PR-2 commits only)
git revert <pr2-commits>
.\scripts\hot-reload.ps1

# Or surgical: disable the new adaptation strategy without code revert
# by setting the env back to old defaults:
#   RETRANSLATION_MAX_ATTEMPTS=10
# (T1 band tightening is in code, not env — needs git revert)
```

PR-1 bug fixes (B1 JSON recovery, B2 ensemble trigger, B3 cap) are
strictly safer than the old behaviour, so we'd keep them on even
in the worst case.

## Optional: episode 143 unhalt

If the operator decides to rerun the WHOLE episode (not just chapter 2)
the `escalated_oscillation` rework_status must be cleared manually:

```powershell
docker exec holodub-postgres-1 psql -U holodub -d holodub `
  -c "UPDATE episodes SET rework_status='' WHERE id=143;"
```

(The rework engine's `runDecisionLoop` skips episodes with halted
status; clearing the field lets the engine re-engage if needed.)
